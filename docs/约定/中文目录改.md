# 中文目录结构建议（完善版）

## 一、总体思路

这套目录结构面向你当前的项目特点设计，重点解决 4 个问题：

1. 原始数据、处理中间数据、最终分析数据分层不清。
2. 脚本、图件、表格、模型结果容易混放。
3. 比赛展示材料和研究过程材料没有明确分区。
4. 后续要接入“单入口流水线”时，缺少稳定目录约定。

因此建议采用“**数据层 - 代码层 - 结果层 - 文档层 - 展示层 - 归档层**”六段结构。

## 三、各部分职责说明

## 1. `data/`

这是数据主目录，按数据生命周期分层。

### `原始数据/`
- 从外部直接获得的数据。
- 包括原始 Excel、CSV、PDF、图片、压缩包。
- 原则上不手动修改。

### `外部资料/`
- 不直接参与建模，但会作为辅助资料使用。
- 例如：
  - `china.geojson`
  - 行政区划底图
  - 标准煤折算系数
  - 公开口径说明文件

### `中间数据/`
- 清洗后但还未最终定稿的数据。
- 包括：
  - 初步合并表
  - 中间拼接结果
  - 临时转换口径版本
  - 测试性 merge 结果

### `最终数据/`
- 可直接进入效率测算、回归分析、空间分析的数据。
- 包括：
  - 最终面板数据
  - 邻接矩阵
  - DEA 输入表
  - 回归使用表

---

## 2. `code/`

这是代码目录。  
建议你以后所有 `.py` 都逐步往这里归。

### `数据处理/`
- 清洗原始数据
- 统一字段名
- 统一省份名、年份格式
- 构建基础面板
- 正式主线中不再保留以排放比例近似 `es` 的代理脚本
- 规则：`code/数据处理/` 下的 `x.py`，内容输出到 `outputs/数据处理/x/` 下

### `效率测算/`
- DEA / SBM / 超效率 SBM
- Malmquist 分解
- 效率结果提取与整理
- 效率测算相关绘图脚本

### `回归分析/`
- 相关性与共线性分析
- 模型设定检验
- 单位根检验
- 基准回归
- 稳健性检验
- 异质性检验

### `空间分析/`
- 邻接矩阵构建
- Global Moran's I
- Local Moran / LISA
- 空间分布图和聚类图

### `流水线/`
- 单入口主脚本
- 分阶段执行脚本
- 运行顺序控制

### `通用工具/`
- 公共函数
- 路径管理
- 字体管理
- 配色方案
- 省份映射和区域映射
- 各类图件作图规范文档

## 补充说明

### 1. 绘图脚本归并规则

- 不再单独保留 `code/可视化绘图/`。
- 与效率结果直接对应的绘图脚本，归入 `code/效率测算/`。
- 与样本构建、缺失检查、变量概览直接对应的绘图脚本，归入 `code/数据处理/`。
- 与回归或空间分析直接对应的图件，归入各自主题目录，不再额外拆出单独绘图目录。

### 2. 回归分析脚本排序规则

- 当前顺序统一为：
  1. 相关性与共线性分析
  2. 模型设定检验
  3. 面板单位根检验
  4. 基准面板回归
  5. 稳健性检验
  6. 异质性检验

### 3. 对话记录命名规则

- `10-60` 号对话记录统一改为 `论文框架1` 至 `论文框架6`。
- 来源明确的文件保留 `cursor_auto`、`codex` 后缀。
- `memory` 文件统一采用 `memoryx_中文主题` 形式，中文主题应直接概括该轮实际工作内容。

### 4. 作图说明文档规则

- 原 `docs/概念解释/` 下的图件说明类文档，统一改名为 `作图规范`。
- 此类文档移入 `code/通用工具/`，作为后续复用绘图口径的辅助资料。

## 3. `outputs/`

这是结果目录，不放脚本，只放脚本跑出来的产物。

### 为什么这里按“分析阶段”分

因为你这个项目不是单一模型，而是有完整链路：

- 数据处理
- 效率测算
- 回归分析
- 空间分析
- 最终导出论文版 / 比赛版

按阶段分结果，比单纯分 `figures / tables / models` 更符合你当前使用习惯，也更容易排查。

### 每个阶段下统一再分
- `图/`
- `表/`
- `模型输出/`
- `日志/`

这样你未来一眼就知道：
- 图去哪找
- 表去哪找
- 模型原始结果在哪
- 运行日志在哪

## 4. `releases/`

这是和 `outputs/` 同级的目录，用于集中存放“最终筛选过、准备展示或写作的结果”，不是所有中间图都往里塞。

### `论文版/`
- 论文正文图
- 论文附表
- 写作正式采用版本

### `比赛版/`
- 视频用图

- 统一风格优化后的展示图

### `备份/`
- 历史版本图件
- 未最终采用的结果
- 临时导出但不想删的文件

## 四、与你当前项目最匹配的迁移原则

### 1. 根目录尽量只保留
- `README.md`
- `requirements.txt`
- `.gitignore`
- 少量总入口脚本
- 目录说明文档

### 2. 当前根目录散落的 `.py`
后续建议逐步移动到 `code/`：

- `build_ceads_energy_es.py` -> `code/数据处理/`
- `extract_dearun_eff.py` -> `code/效率测算/`
- `correlation_vif_analysis.py` -> `code/回归分析/`
- `global_morans_i_codex.py` -> `code/空间分析/`
- `gm_decomp_plots.py` -> `code/效率测算/`

### 3. 当前 `prcd/` 中的结果目录
后续建议逐步归入 `outputs/` 对应阶段下。

例如：
- `eff_plots/` -> `outputs/效率测算/图/`
- `baseline_panel_outputs/` -> `outputs/回归分析/`
- `spatial_plots/` -> `outputs/空间分析/`
- `correlation_plots/` -> `outputs/回归分析/图/`

### 4. 当前 `memory*.md`
建议统一放入：

- `docs/对话记录/`

### 5. 当前 `修改意见.txt`
建议放入：

- `docs/待解决问题/`

## 五、命名建议

## 1. 目录命名

- 一级和二级目录可以中英混合，保持稳定。
- 三级及以后目录中文化，便于你自己识别。
- 比赛展示材料统一进入 `presentation/`。

## 2. 脚本命名

建议脚本文件名仍然保留英文，便于运行：

- `build_*.py`
- `run_*.py`
- `plot_*.py`
- `export_*.py`

## 3. 数据命名

建议体现主题、年份和版本：

```text
panel_base_2015_2022_v1.csv
panel_analysis_2015_2022_v1.csv
efficiency_dearun_2015_2022_v1.csv
```

## 4. 图件命名

建议体现用途：

```text
regression_coefficient_forest_contest.png
spatial_global_moran_paper.png
efficiency_rank_map_contest.png
```

## 六、最终建议

如果你希望这套目录以后还能接上“单入口流水线”，那最稳的做法是：

1. `data/` 只按数据生命周期分层。
2. `code/` 只放代码。
3. `outputs/` 只放结果，并按分析阶段分层。
4. `docs/` 只放说明和记录。
5. `presentation/` 只放比赛展示材料。
6. `archive/` 统一收纳历史和临时文件。

这样做的好处是：

- 你自己找文件会快很多；
- 后续写论文、做视频、做答辩时不会反复翻目录；
- 真要做自动流水线时，目录已经天然支持。
